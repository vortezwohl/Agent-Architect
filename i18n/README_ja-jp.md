## 問題

ほとんどのユーザーは、エージェントにアーキテクチャの判断を求めているわけではありません。

通常の作業を依頼しています。

```text
請求書をダウンロードするエンドポイントを追加する。
設定ページを作成する。
このサービスを新しいプロバイダーに接続する。
このモジュールをリファクタリングする。
この機能を拡張可能にする。
```

エージェントは最初の一行を書く前から、すでに次のことを決め始めています。

- どのモジュールが振る舞いと状態を所有するか。
- どの依存関係が境界をまたぐか。
- どの失敗経路が公開される振る舞いになるか。
- 直接実装、抽象化、フレームワーク拡張のどれに根拠があるか。
- 変更をどのように発展、移行、ロールバック、検証するか。

これらはアーキテクチャ上の判断です。

明示的なプロトコルがなければ、エージェントは通常、次の三つの方法で失敗します。

<table>
<tr>
<th align="left">失敗</th>
<th align="left">何が起こるか</th>
</tr>
<tr>
<td><strong>偶然のアーキテクチャ</strong></td>
<td>エージェントは最も近いファイルに機能を直接書き込みます。誰も境界に名前を付けないうちに、依存関係、状態、エラー経路が広がります。</td>
</tr>
<tr>
<td><strong>推測による抽象化</strong></td>
<td>エージェントはあり得る未来を見て、実際の変化が現れる前にインターフェース、ファクトリー、レジストリ、イベント、継承を作成します。</td>
</tr>
<tr>
<td><strong>宣言だけの検証</strong></td>
<td>振る舞い、互換性、失敗処理、残るリスクを証明せずに、リファクタリングがきれい、または完了したと宣言します。</td>
</tr>
</table>

**Agent Architect** は、非自明なコーディング要求と構造的な複雑さの間に置くガードレールです。

エージェントに現実を調査させ、根拠のある最小設計を選ばせ、複雑さが広がる前に証拠を報告させます。

## 違いを見る

### Agent Architect を使う前

```text
ユーザー: 「請求書をダウンロードするエンドポイントを追加して。」

エージェント: ハンドラーを書き、永続化層に直接入り、ストレージを呼び出し、
認可をインラインで追加し、「将来の形式」のために ExportFactory を作成して、
そのエンドポイントは完成したと報告する。
```

### Agent Architect を使う場合

```text
ユーザー: 「請求書をダウンロードするエンドポイントを追加して。」

エージェント:
1. 既存の請求書の所有権、認可、ストレージ、呼び出し元、
   エラー処理、テストを調査する。
2. 実際の変更を明示する。既存の請求書成果物に対する HTTP の入口を一つ追加する。
3. アプリケーションサービスへの直接呼び出しと新しいエクスポート層を比較する。
4. 形式が一つでストレージ経路も一つなら、直接設計を維持する。
5. 抽出のトリガーを記録する。同じアプリケーション操作の背後で、独立して変化する複数の
   エクスポート形式またはストレージプロバイダーがある場合。
6. 認可、請求書欠落、ストレージ障害、既存の請求書フローを検証する。
```

結果は、エンジニアリングが少なくなることではありません。

それは**明確な判断、より小さい影響範囲、そして検証経路を持つエンジニアリング**です。

## 決定プロトコル

```text
非自明なコーディング要求を受け取る
      |
      v
リポジトリと現在の振る舞いを調査する
      |
      v
証明された変更、安定した中核、制約、失敗経路を明示する
      |
      +-- 独立した変化の証拠がないか？
      |     `-- 直接設計を維持し、抽出のトリガーを記録する。
      |
      v
最小の直接解と候補となる構造を比較する
      |
      v
不要なインターフェース、ファクトリー、ラッパー、イベント、継承を除外する
      |
      v
所有権、ライフサイクル、API の方向、トランザクション、並行性、
ロールバック、可観測性、フレームワークの慣習を確認する
      |
      v
最小で検証済みの一つのスライスを実装する
      |
      v
証拠、決定、実行した検証、残るリスクを報告する
```

### 譲れないルール

- 何が独立して変化し、何が安定しているかを明示する前に、構造を生成しないでください。
- 分離する具体的な変化と追加するコストを示さずに、インターフェース、ファクトリー、ラッパー、イベント、継承階層、グローバルオブジェクト、フレームワーク層を導入しないでください。
- 不確実性を推測的な抽象化に変えないでください。
- 配送セマンティクスなしにイベントを非同期設計と呼ばないでください。
- 依存性注入を避けるために Singleton を使わないでください。
- コンポジション、コールバック、直接関数の方が明確なら、継承を使わないでください。
- 未検証の実装やリファクタリングを完成と呼ばないでください。

## エージェントが生成するもの

非自明な機能、統合、設計、リファクタリングごとに、Agent Architect は監査可能な記録を生成します。

```text
01. 設計診断
    目的、非目標、リポジトリの証拠、呼び出し元、安定した中核、
    変化点、臭い、制約。

02. 代替案
    最小の直接設計、候補となる構造、
    却下した代替案、継続する構造コスト。

03. 決定
    選択した設計、公開 API と依存関係への影響、
    移行とロールバックへの影響、またはパターンを使わない根拠ある決定。

04. 検証
    正常、境界、失敗、統合、運用の確認；
    実際に実行した検証；残る不確実性またはリスク。
```

## 使用する場面

次を含む**あらゆる非自明な変更を実装またはレビューする前に** Agent Architect を使用してください。

- モジュール、レイヤー、サービスをまたぐ新機能。
- 依存関係、所有権、状態、ライフサイクルを変えるリファクタリング。
- サードパーティ統合とレガシー適応。
- 非同期、イベント、トランザクション、リトライ、キャッシュ、認可に関する作業。
- システムを「きれいに」「拡張可能に」「スケーラブルに」「将来に備えて」ほしいという要求。
- コードは妥当に見える一方で、構造上の判断が暗黙的な PR レビュー。

> [!TIP]
> ユーザーがアーキテクチャの質問をするまで待たないでください。機能作業と構造的変更を扱うエージェントのワークフローに、このスキルを追加してください。

## パターンは製品ではなく道具です

Agent Architect は 23 種すべての GoF パターンを扱いますが、パターン一覧から始めるわけではありません。

証拠から始めます。**何が変わるか、誰が所有するか、何が失敗するか、何を安定させるべきか。**

<details>
<summary><b>生成に関する決定</b></summary>

- Factory Method
- Abstract Factory
- Builder
- Prototype
- Singleton scope

</details>

<details>
<summary><b>構造に関する決定</b></summary>

- Adapter
- Bridge
- Composite
- Decorator
- Facade
- Flyweight
- Proxy

</details>

<details>
<summary><b>振る舞いに関する決定</b></summary>

- Chain of Responsibility
- Command
- Interpreter
- Iterator
- Mediator
- Memento
- Observer
- State
- Strategy
- Template Method
- Visitor

</details>

<details>
<summary><b>重要なパターン境界</b></summary>

| 混同しないもの | 対象 |
| --- | --- |
| Decorator | Proxy または Adapter |
| Facade | Mediator |
| Factory Method | Abstract Factory、Builder、または Prototype |
| Strategy | State または Template Method |
| Observer | Chain of Responsibility または Command |
| Composite | Decorator |

パターンは**意図、協力者、ライフサイクル、変化、失敗時の振る舞い**によって選びます。クラス図だけで選ぶことは決してありません。

</details>

## 基準

機能はコンパイルできるから完成するのではありません。

リファクタリングはコードがよりきれいに見えるから完成するのではありません。

アーキテクチャ上の判断は、次に答えられる場合にのみ完了です。

```text
どの証拠がこの構造を正当化したか？
なぜ直接の代替案では不十分か？
何が独立して変化するか？
何を却下し、なぜか？
振る舞いはどのように検証されたか？
どのリスクが残っているか？
```

> **パターン名のためではなく、次に検証される変更のために設計する。**

---

## コントリビューション

貢献は**判断力**を向上させるべきであり、儀式を増やすべきではありません。

有用な貢献には次が含まれます。

- 実際の機能とアーキテクチャ判断に対する証拠ゲート。
- より明確なパターン境界と却下基準。
- 偶然のアーキテクチャと推測的抽象化を再現可能にする例。
- ライフサイクル、並行性、トランザクション、移行、ロールバックに関する検証ガイダンス。
- エージェントが構造的複雑さを広げにくくする修正。

変更を作成する前に、次を問いかけてください。

```text
これは、コードが広がる前にエージェントが行える判断を改善するか？
この改善は検証できるか？
推測的なプロセスを増やさずにガイダンスを加えるか？
```
